# KeeperFX 二次开发日志

**项目名称**: KeeperFX - 界面语言与语音语言分离功能
**开发者**: ShenyfZero9211
**日期**: 2026-04-02

---

## 1. 核心需求分析
在原版 KeeperFX 中，界面的语言（UI）和讲解语音（Speech）是通过 `LANGUAGE` 指令绑定的。例如设置 `LANGUAGE = CHI`，则界面和语音都会切换为中文。
**目标**：解耦二者，实现“中文界面 + 英文语音”的灵活组合。

## 3. 编译环境攻坚（重点）

由于项目基于 32 位 (i686) 架构，在现代 64 位 Windows 环境下配置编译链遇到了诸多挑战：

### 3.1 架构冲突
- **现象**: 使用 64 位 MinGW 时，由于 `bflib_crash.c` 涉及底层寄存器寄存器操作（`Rip` vs `Eip`），导致编译报错。
- **对策**: 确定使用 **32 位 MinGW (i686)** 系列工具链。

### 3.2 资源定义冲突 (RT_VERSION)
- **现象**: 链接阶段报错 `duplicate leaf: type: 10 (VERSION)`。
- **原因**: KeeperFX 自身的 `VERSIONINFO` (ID 1) 与链接的第三方库资源冲突。
- **对策**: 将 `res/keeperfx_stdres.rc` 中的版本资源 ID 修改为 **2**，成功避开 ID 1 的全局冲突。

### 3.3 线程模型匹配 (thread model)
- **现象**: 链接阶段出现大量 `undefined reference to __gthr_win32_mutex_unlock`。
- **原因**: 默认下载的 WinLibs 指令集使用 POSIX 线程模型，而 KeeperFX 的二进制库（如 Luajit/OpenAL）使用 Win32 线程模型。
- **最终对策**: 使用 **GCC 13.2.0 (i686-win32-dwarf)** 版本的套件。其内置的 Win32 线程原语与项目库完美兼容。

---

## 4. 最终编译方案总结

1. **工具链**: D:\mingw32 (GCC 13.2.0, Win32 threads).
2. **构建环境**: 
   - 映射盘符 `M:` 到 MinGW 目录（规避路径空格问题）。
   - 环境路径: `M:\bin;C:\Program Files\Git\bin;...`
3. **指令**: `mingw32-make.exe -j4 all`
4. **产物**: `bin/keeperfx.exe`
